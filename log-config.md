# Extracting Actionable Gameplay Data from Hearthstone Logs

## Introduction

Hearthstone provides detailed log files (when verbose logging is enabled) that can be parsed to extract key gameplay data for analytics and visualization. The primary logs of interest are **Power.log** (which records game state changes for all modes) and **Bob.log** (which records Battlegrounds shop-phase events, named after Bartender Bob). By analyzing these logs, developers can programmatically determine outcomes and in-game actions such as wins/losses, heroes used, match results (including Battlegrounds placements), and various in-game decisions. This report outlines the most meaningful data points available from Hearthstone logs, how to extract them reliably, and how they support analytics or visualizations. Known limitations (e.g. the need for verbose logging, or missing fields like MMR) and quirks (like duplicated log entries and timing issues) are also addressed. The focus is on information useful for building a CLI or backend parser that processes match data.

## Game Outcomes: Wins, Losses, and Placement

**Log Source:** _Power.log_  
**Key Patterns:** Look for `TAG_CHANGE` events on player entities with the tag `PLAYSTATE`. At match conclusion, the game sets each player's `PLAYSTATE` to a terminal value (`WON`, `LOST`, or `TIED`). For example, a log line might read:

```
TAG_CHANGE Entity=YourPlayerName tag=PLAYSTATE value=WON
```

while your opponent’s corresponding entry would be `value=LOST`. In the raw Power.log, it appears as in this snippet (showing one player losing and the other winning) ([D 20:52:07.0158503 GameState.DebugPrintPower() - ACTION_START Entity=[name=Sabot - Pastebin.com](https://pastebin.com/EBjhrEMA#:~:text=Entity%3DPatashu%20tag%3DPLAYSTATE%20value%3DLOST)):

> _TAG_CHANGE Entity=Patashu tag=PLAYSTATE value=LOST_  
> _TAG_CHANGE Entity=NightKev tag=PLAYSTATE value=WON_

Here, the `Entity` field identifies the player by name (or internal ID) and the `PLAYSTATE` tag indicates the outcome. A parser can reliably detect a win or loss by scanning for `value=WON` or `value=LOST` on the local player's entity. Typically, you determine which entity is the local player earlier in the log (see **Game Start Events** below), then check that entity’s final PLAYSTATE. Many community tools use exactly this approach – for instance, Track-o-Bot’s source code looks for `tag == "PLAYSTATE"` with values `"WON"`, `"LOST"`, or `"TIED"` to trigger end-of-game outcome handling ([track-o-bot/src/HearthstoneLogTracker.cpp at master · stevschmid/track-o-bot · GitHub](https://github.com/stevschmid/track-o-bot/blob/master/src/HearthstoneLogTracker.cpp#:~:text=if%28%20tag%20%3D%3D%20,%29%20%29)).

**Extraction:** Programmatically, you can use regex on Power.log lines or a log parsing library to find `TAG_CHANGE` lines containing `tag=PLAYSTATE`. Check the `value` and which player/entity it refers to. Since each match will have exactly one winner (or a tie), the local player’s PLAYSTATE is the primary indicator of a win or loss. Emitting an event when this appears is a common practice – for example, the **hearthstone-log-watcher** module raises a `game-over` event with the players and their status once it sees these tags ([hearthstone-log-watcher - npm](https://www.npmjs.com/package/hearthstone-log-watcher/v/1.0.1#:~:text=game)). The log watcher’s output example shows each player’s name with a status of 'WON' or 'LOST' at game end ([hearthstone-log-watcher - npm](https://www.npmjs.com/package/hearthstone-log-watcher/v/1.0.1#:~:text=Exmample%20player%20object%3A)).

**Analytics Use:** Win/loss data enables calculation of win rates, streaks, and match outcomes over time. This can feed into visualizations like a line chart of win-rate over the last N games, or a summary dashboard showing total wins, losses, and win percentage. Combined with other data (hero or deck used, opponent info), it can produce insightful breakdowns (e.g. win rate per hero class or per deck archetype).

**Note:** If verbose logging is not enabled for the Power log, these detailed tag changes may not appear. Ensure the Hearthstone **log.config** is set to `LogLevel=Verbose` for the Power category. Another quirk is that the Power.log often logs duplicate events from both `GameState.DebugPrintPower()` and `PowerTaskList.DebugPrintPower()` – meaning certain actions (including final tag changes) might appear twice or interleaved ([hearthstone-battlegrounds-simulator/src/log_parser.cpp at master · twanvl/hearthstone-battlegrounds-simulator · GitHub](https://github.com/twanvl/hearthstone-battlegrounds-simulator/blob/master/src/log_parser.cpp#:~:text=%2F%2F%20log%20line%20type)). A robust parser should handle or filter out duplicate `TAG_CHANGE` entries. Also, Hearthstone does not log post-match ratings for Battlegrounds or MMR changes in the Power.log; only the win/loss/placement is logged ([Can not find hero data in hearthstone battleground log file - Technical Support - Hearthstone Forums](https://us.forums.blizzard.com/en/hearthstone/t/can-not-find-hero-data-in-hearthstone-battleground-log-file/25592#:~:text=,mmr)).

## Hero Played (Class or Battlegrounds Hero)

**Log Source:** _Power.log_  
**Key Patterns:** Identify the hero card entity for both the local player and the opponent. In constructed modes (Standard, Arena, etc.), this corresponds to the class hero (e.g. Jaina for Mage, Rexxar for Hunter). In Battlegrounds, it’s the chosen Battlegrounds hero (e.g. “Millhouse Manastorm” or “Archvillain Rafaam”). The logs do not explicitly say “Hero X was chosen”; instead, the information is encoded in the game state. After the initial mulligan/hero selection phase, each player’s **Player entity** will have a tag `HERO_ENTITY` pointing to the hero’s entity ID. One can find a line like:

```
TAG_CHANGE Entity=[name=YourPlayer] tag=HERO_ENTITY value=<entity_id>
```

This appears after hero selection (or after mulligan in a constructed game). To get the actual hero, you then find the `FULL_ENTITY` definition of that `<entity_id>` which will include the `CardID` of the hero card ([Can not find hero data in hearthstone battleground log file - Technical Support - Hearthstone Forums](https://us.forums.blizzard.com/en/hearthstone/t/can-not-find-hero-data-in-hearthstone-battleground-log-file/25592#:~:text=,play)). For example, if `HERO_ENTITY value=64`, you would find the entity with `id=64` earlier in the log; its `CardID` might be `HERO_08` (which corresponds to Jaina Proudmoore) or something like `TB_BaconShop_HERO_22` (a Battlegrounds hero). The forum guidance confirms this process: _“See what the HERO_ENTITY tag on your player entity is set to after the mulligan. Then go find the card ID of that entity.”_ ([Can not find hero data in hearthstone battleground log file - Technical Support - Hearthstone Forums](https://us.forums.blizzard.com/en/hearthstone/t/can-not-find-hero-data-in-hearthstone-battleground-log-file/25592#:~:text=,play)).

In many cases, the creation of hero entities is also logged. For constructed, you might see a `FULL_ENTITY - Creating` line for each hero at game start, e.g. `FULL_ENTITY - Creating [id=64 cardID=HERO_08]`. In Battlegrounds, hero selection happens in a pre-game phase, but once the game proper begins, the chosen hero is assigned to the player’s HERO_ENTITY. Sometimes the log will show an update like `FULL_ENTITY - Updating [...] CardID=TB_BaconShop_HERO_22` when the hero is confirmed ([hearthstone-battlegrounds-simulator/src/log_parser.cpp at master · twanvl/hearthstone-battlegrounds-simulator · GitHub](https://github.com/twanvl/hearthstone-battlegrounds-simulator/blob/master/src/log_parser.cpp#:~:text=if%20%28in.match_exact%28)). By tracking these, you can capture exactly which hero was used.

**Extraction:** Programmatically, maintain a mapping of player entity to hero. When parsing the Power.log for a new game, first capture lines starting with `CREATE_GAME` (to know a game started) and then find `Player` entities and their tags. One approach is to wait until the mulligan choices are done (in constructed) or hero selection is done (in BG), then look for `tag=HERO_ENTITY` on the friendly player. Alternatively, you can parse every `FULL_ENTITY`/`SHOW_ENTITY` for a card of type HERO assigned to a player. Logging libraries like **HSReplay’s hslog** build an entity tree where each player entity has a HERO_ENTITY tag linking to the hero object ([Can not find hero data in hearthstone battleground log file - Technical Support - Hearthstone Forums](https://us.forums.blizzard.com/en/hearthstone/t/can-not-find-hero-data-in-hearthstone-battleground-log-file/25592#:~:text=,play)). Once you have the hero card ID, you can map that to a hero name or class using a static lookup (Blizzard’s card IDs are consistent: e.g. HERO_08 → Jaina/Mage).

For the opponent’s hero or class, the process is similar: find the opponent player’s HERO_ENTITY tag or observe the other hero entity created. Often, the log shows both players’ heroes being created near game start. You can also detect opponent class by seeing the opponent hero cardID (e.g. HERO_05 means the opponent is Rexxar, a Hunter).

**Analytics Use:** Knowing which hero was played is crucial for context. In constructed modes, this translates to class – enabling winrate by class, or matchup statistics (e.g. how often you win as Mage versus Warrior). In Battlegrounds, tracking the hero choice allows analytics on average placement per hero, which heroes you perform best with, etc. Visualizations might include a bar chart of win rates per class, or an aggregation of Battlegrounds placements by hero. It also enables filtering of other stats (you could filter all other metrics by the hero or class played).

**Limitations:** The hero choice in Battlegrounds is _not_ explicitly listed in the log; it must be derived via the HERO_ENTITY tag as described. This is a known quirk that developers have pointed out (the logs were “missing” direct hero info) ([Can not find hero data in hearthstone battleground log file - Technical Support - Hearthstone Forums](https://us.forums.blizzard.com/en/hearthstone/t/can-not-find-hero-data-in-hearthstone-battleground-log-file/25592#:~:text=I%20have%20analyzed%20the%20log,some%20important%20things%20were%20missing)). Additionally, the hero’s name might not be directly in the log; you often get a card ID and will need a card database to get the human-readable name or class. Ensure the parser is equipped with a mapping of card IDs to hero names/classes (Blizzard’s APIs or the game data can provide this).

## Match Outcome and Battlegrounds Placement

**Log Source:** _Power.log_  
**Key Patterns:** In 1v1 modes, the match outcome is fully captured by the win/loss tags discussed above. For Battlegrounds (8-player free-for-all), the outcome is each player’s **placement** (1st through 8th). Hearthstone logs a tag on each player for their final standing in BG called `PLAYER_LEADERBOARD_PLACE`. When a Battlegrounds game ends (or when a particular player is eliminated), you will see a `TAG_CHANGE` on that player’s entity with `tag=PLAYER_LEADERBOARD_PLACE value=N` (where N=1 for first place, 2 for second, etc.) ([Can not find hero data in hearthstone battleground log file - Technical Support - Hearthstone Forums](https://us.forums.blizzard.com/en/hearthstone/t/can-not-find-hero-data-in-hearthstone-battleground-log-file/25592#:~:text=,mmr)). The local player’s placement can be determined by watching for this tag on the friendly player entity. For example, if you finish 4th, expect a line like:

```
TAG_CHANGE Entity=[name=YourPlayer] tag=PLAYER_LEADERBOARD_PLACE value=4
```

If you win the lobby, you’d see value=1 on your entity. These tags typically appear at game conclusion for all players (or upon each player’s elimination).

**Extraction:** Similar to wins, scan the Power.log for `PLAYER_LEADERBOARD_PLACE` tag changes involving the local player. Usually, this will come at the very end of the match (if you got 1st) or at the moment you died (for lower placements). A robust approach is to monitor all tag changes for `PLAYER_LEADERBOARD_PLACE` on any player, and record the one for the local player. If using a structured parser, the local player’s entity will have a tag dictionary – you can simply read `player.tags[PLAYER_LEADERBOARD_PLACE]` after the game ends. Keep in mind that if your parser reads after the game is over, it might catch the final state with that tag set.

**Analytics Use:** Placement data powers many Battlegrounds-specific metrics. You can calculate average placement, top-4 rate (how often you finish in the top half), distribution of finishes (e.g. pie chart of 1st/2nd/…/8th frequency), etc. It’s also useful per hero: e.g. average placement when playing as each BG hero. Visualization examples include bar graphs of placement frequency or a leaderboard-style summary of one’s performance.

**Note:** As mentioned earlier, **MMR changes or rating** are _not_ included in the logs ([Can not find hero data in hearthstone battleground log file - Technical Support - Hearthstone Forums](https://us.forums.blizzard.com/en/hearthstone/t/can-not-find-hero-data-in-hearthstone-battleground-log-file/25592#:~:text=,mmr)). If you need the rating gain/loss, you must obtain it via other means (screen-scraping or Blizzard’s post-game APIs, if any). The logs also don’t directly tell you which place counts as a “win” for Battlegrounds (generally 1st–4th are considered wins for daily quests and rating gains ([Battlegrounds - Hearthstone Wiki - Fandom](https://hearthstone.fandom.com/wiki/Battlegrounds#:~:text=Battlegrounds%20,of%20the%20composition%20of)), but the log itself just gives the numeric placement). The parser may need to interpret placements 1-4 as “won” in a broad sense if that’s relevant to your analytics.

## Battlegrounds-Specific Metrics (Shop Phase Data)

Battlegrounds games involve unique mechanics not present in constructed matches. The **Bob.log** (enabled via verbose logging for the “Bob” category) contains events from the shop (recruit) phase, such as tavern refreshes and minion offers. Combined with Power.log (which still logs combat and entity state), these allow extraction of rich data about the player’s decisions and resources. Below are key BG data points and how to get them:

### Number of Rerolls (Tavern Refreshes)

**Log Source:** _Bob.log_ (primary) and _Power.log_ (secondary)  
**Key Patterns:** A “reroll” occurs when the player spends 1 Gold to refresh Bob’s Tavern offerings. In Bob.log, this often corresponds to a line indicating the shop was refreshed. For example, Bob’s dialog or a debug line might be printed each time you hit the refresh button. While an exact sample log line is not readily documented publicly, you might see phrases like “Refreshing Tavern” or Bob offering new minions. If Bob.log is verbose, each refresh could trigger a listing of the new minions in the shop (effectively showing the result of the reroll). In the absence of Bob.log, you can infer rerolls by tracking gold usage in Power.log: each manual refresh costs 1 Gold, so if you notice the `RESOURCES` (gold) count dropping by 1 outside of the free start-of-turn refresh, that indicates a reroll.

**Extraction:** The most straightforward method is counting occurrences of a shop refresh event in Bob.log. For instance, if Bob.log prints the shop minions whenever they change, count how many times that happens beyond the initial roll each turn. Another approach is to observe the **Tavern Tier upgrade timing** vs. shop refreshes. Normally, each turn begins with an automatic refresh (free) – any _additional_ refresh that turn is player-initiated. You could maintain a counter that resets each turn: every time new shop minions appear beyond the first, increment a reroll count. In Power.log, monitor the player’s gold (`RESOURCES` tag): whenever it decreases by exactly 1 without a purchase or level-up, it likely corresponds to a reroll. Also, Hearthstone may use a tag like `NUM_REFRESHES_USED` (not confirmed in public docs) or simply rely on the gold deduction. A safe bet is to rely on Bob.log text if available.

**Analytics Use:** Counting rerolls allows analysis of how efficiently a player is using their gold and how aggressively they are searching for specific minions. For example, one could graph the number of rerolls per turn or per game, or correlate high reroll counts with final placement. An analytics tool might highlight if a player rerolls too much (wasting gold) or too little (perhaps missing opportunities to find key minions). Visualization could be as simple as a total rerolls per game metric, or a turn-by-turn breakdown in a timeline.

**Note:** Ensure Bob.log is enabled (`[Bob] LogLevel=Verbose` in log.config). Without it, detecting rerolls purely from Power.log is possible but more error-prone (you’d be inferring from gold usage). Also note that if an anomaly or hero power provides a free refresh, it might appear as a shop refresh without a gold decrease – a parser should consider game context (some heroes have mechanics affecting refreshes). Generally, however, manual refresh actions are clearly logged in Bob.log when verbose.

### Gold Spent and Economy Usage

**Log Source:** _Power.log_ (for gold resource tags) and _Bob.log_ (for shop actions)  
**Key Patterns:** In Battlegrounds, gold is essentially the “mana” of the mode. The Power.log uses the same resource tags for gold: `RESOURCES` represents current gold, and `RESOURCES_USED` how much has been spent so far in the turn (these tags reset each turn). For example, when a turn starts, `RESOURCES` might be set to 10 (if it’s turn 10 of BG), and as you spend gold, you will see `TAG_CHANGE` events like `RESOURCES_USED value= X`. Each purchase (3 gold) or tavern tier upgrade (varies by tier) or reroll (1 gold) increments that count. The log snippet below (from constructed) illustrates how the engine logs resource usage ([D 20:52:07.0158503 GameState.DebugPrintPower() - ACTION_START Entity=[name=Sabot - Pastebin.com](https://pastebin.com/EBjhrEMA#:~:text=2.%20D%2020%3A52%3A07.0178504%20GameState.DebugPrintPower%28%29%20,TAG_CHANGE%20Entity%3DPatashu%20tag%3DRESOURCES_USED%20value%3D4)), and a similar pattern occurs in BG:

> _TAG_CHANGE Entity=Patashu tag=RESOURCES_USED value=4_ ([D 20:52:07.0158503 GameState.DebugPrintPower() - ACTION_START Entity=[name=Sabot - Pastebin.com](https://pastebin.com/EBjhrEMA#:~:text=2.%20D%2020%3A52%3A07.0178504%20GameState.DebugPrintPower%28%29%20,TAG_CHANGE%20Entity%3DPatashu%20tag%3DRESOURCES_USED%20value%3D4))

In BG, if you bought two minions (6 gold) and did one reroll (1 gold) in a turn, you’d expect a `RESOURCES_USED value=7` by the end of that turn on your player. Bob.log might also explicitly log purchases or upgrades (e.g., a line like “Upgraded to Tavern Tier 4 for 7 Gold” or “Bought [MinionName] for 3 Gold”), though the exact text varies.

**Extraction:** Tallying gold spent can be done by summing all expenditures each turn. The simplest method: for each turn, read the final `RESOURCES_USED` for the player – that is the gold spent that turn. Sum those over the whole game to get total gold spent. You could also increment a counter on each action: +3 for a minion buy, +1 for reroll, +[cost] for tavern upgrade. These actions can be identified via Bob.log or via tag changes. Notably, tavern upgrades are indicated by the `PLAYER_TECH_LEVEL` tag increasing (this tag represents Tavern Tier). When you see `PLAYER_TECH_LEVEL` change from N to N+1 on the player, you know an upgrade was bought for a known cost (which is standard per tier: e.g. Tier 3→4 costs 7 gold). A parser might calculate gold spent on upgrades from those events or simply rely on the resource tags.

Another reliable source is the `RESOURCES` tag itself: it will decrement with each spend. By tracking the `RESOURCES` value throughout the turn (or comparing it before and after an action), one can deduce how much was spent on that action. In verbose logs, every purchase or reroll should trigger a resource update. For instance, buying a minion might produce: `TAG_CHANGE Entity=YourPlayer tag=RESOURCES value= (previous_gold-3)` as well as `RESOURCES_USED` incremented by 3.

**Analytics Use:** Gold usage patterns are crucial in BG analysis. You can compute metrics like “gold efficiency” – how much gold is unspent at end of turns (ideally 0 most of the time). A visualization could be a turn-by-turn bar chart of gold spent vs. gold available, highlighting any leftover gold (which usually indicates inefficiency). Another useful chart is cumulative gold spent on rerolls vs. on minion purchases/upgrades. Since Bob.log can tell exactly what gold was spent on (if parsed carefully), one could categorize gold spend: e.g. X% on minions, Y% on rerolls, Z% on tavern upgrades. This helps identify strategic tendencies (maybe a player spends too much on rolling and not enough on upgrading, etc.).

**Note:** As with other data, missing verbose logs will hinder this – without resource tags, you can’t track gold. Also be aware of special cases: some heroes have gold-related mechanics (e.g. Captain Hooktusk gets a free minion sell ability, etc.) that could skew simple counts. But the log’s resource counters will still reflect net gold usage. Timing-wise, Power.log entries for gold changes might be slightly out of order relative to Bob.log (shop) events, but using timestamps or processing both logs in tandem (they share timestamps) can synchronize the data.

### Shop Offerings (Minions Offered by Bob)

**Log Source:** _Bob.log_ (primary), _Power.log_ (for backup analysis)  
**Key Patterns:** Each round in Battlegrounds, Bob’s Tavern offers a set of minions for purchase. Capturing these offerings can provide insight into the options the player had. With verbose Bob logging, the client will list the cards presented in the shop. Typically, when the tavern is populated or refreshed, Bob.log will output the list of minions (likely by card name or ID). For example, you might see lines enumerating the tavern’s contents such as: “Bob offers: [Scallywag], [Micro Machine], [Bronze Warden]” (the exact format could differ, but the idea is that the offered minion names or IDs are shown). Every time the shop changes – start of turn, after a reroll, after buying a minion (which immediately replaces that slot with a new minion from the pool if you haven’t frozen) – the log should reflect the new state.

If Bob.log is not available, Power.log still contains this info but more obliquely: the shop minions are game entities as well. They might appear in the log as entities in the _“SHOP” zone or on the board controlled by Bob_. In the game engine, Bob’s Tavern minions could be represented by an entity with tags indicating they are in the shop. For instance, when a new minion appears in the shop, a `FULL_ENTITY` creation might occur with a card ID of that minion and a tag like `ZONE = SHOP`. (The exact tag for shop isn’t documented in the public wiki, but it’s conceptually a zone). As an example, a creation line could be: `FULL_ENTITY - Creating [id=72 cardID=BG20_301 player=2 zone=HAND]` – notional example – where player=2 might be Bob and zone=HAND might double for tavern. However, because this is complex, relying on Bob.log’s direct printout is easier.

**Extraction:** When Bob.log is verbose, simply parse lines for known minion names or card IDs. You might maintain a list that resets each refresh. For each “Shop update” event, record the offered minions (there are typically 3 to 7 minions depending on tavern tier and game stage). One approach is to detect a marker in Bob.log that indicates the listing (perhaps the presence of card IDs or a specific format) and then parse the line to extract each minion. In Power.log, alternatively, you could listen for `BLOCK_START` events that indicate the beginning of the recruit phase or use the fact that at the start of each recruit phase, the game will create entities for each offered minion. A clever trick is to track `ZONE` changes: when the game transitions from combat to recruit, a bunch of minions move from “removed” into play (the shop). Those appear as creations or moves that you can capture.

**Analytics Use:** Having the full record of what was offered each turn is invaluable for decision analysis. You can answer questions like: “Did the player skip useful minions?” or “How lucky were the shop rolls?”. It enables visualizations such as a turn-by-turn list or table of offered minions, maybe highlighting which ones were bought or frozen. Over many games, one could aggregate the frequency of certain key minions offered by turn (for luck metrics) or how often a player finds triples. If the tool is for personal improvement, showing the user _“these were all your options, and here’s what you chose each turn”_ is a powerful replay aid.

**Note:** The main limitation is volume – listing every shop offering every turn generates a lot of data. Storing and visualizing it requires careful UI design to not overwhelm. Also, there might be edge cases (like when a triple occurs, you get an extra discover; those discovers are not Bob’s normal shop and might appear in a different log or as a `Discover` block in Power.log). Those should be handled separately. Additionally, if the log is not verbose, you won’t get this data at all – normal logging won’t list shop minions to avoid giving players an advantage via external tools, so enabling verbose is a must (with user’s understanding that it might impact performance slightly).

### Final Warband Composition

**Log Source:** _Power.log_  
**Key Patterns:** The “warband” refers to the set of minions on a player’s board. We want the final warband composition – essentially the minions (and possibly their stats or golden status) that the player ended the game with. This could mean the minions on board at the moment the game ended (if you got first place) or the last board you had before you were eliminated (if you died earlier). The Power.log contains full information about minions on the board, since it tracks all entities in play. Toward the end of a game (final combat), all your minions in play are listed as entities with `zone=PLAY` under your control. If you won the last combat, those minions will remain until the victory is declared. If you lost, technically those minions might have died in combat just before elimination – but we can still capture them by looking at the game state at the start of that final combat round.

Concretely, one strategy is to capture the player’s board at the start of each combat phase. Hearthstone’s log delineates phases with the `BLOCK_START` of type `ATTACK` or `TRIGGER` etc., or with the `STEP` tag progression. For example, when the recruit phase ends and combat begins, the game’s `STEP` might change from `MAIN_START` (recruit) to `MAIN_COMBAT` or similar. Just as combat starts, whatever minions have `ZONE = PLAY` for the player constitute that round’s warband. The final warband is simply the last such set recorded. This will include which minions, and if needed one can also read their tags (like ATK, HEALTH, DIVINE_SHIELD etc.) to know their stats and whether they are golden (golden minions have the `PREMIUM` tag set to 1).

**Extraction:** Use the Power.log to track minion entities on the board. One way is to listen for `ZONE` change events or `FULL_ENTITY` creations for your minions. When a minion is played or appears (e.g. via a triple) during the recruit phase, it will move to the play zone (the board). The log will show something like `ZONE_CHANGE entity=[minion] from FRIENDLY HAND to FRIENDLY PLAY`. A BG-specific parser can maintain a list of friendly board minions that updates with these zone changes (similar to how deck trackers handle the board state). Then, at game end, query that list. However, to ensure it's the final composition _at its peak_, you should capture it right before combat or at the moment of elimination.

A robust approach: every time combat starts (you can detect the transition by a log line like **“Begin Combat Phase”** or the first attack), copy the current friendly board minions into a “snapshot”. When the game ends, use the latest snapshot as the final warband. This accounts for the case where the player loses and all minions die in that combat (they would have vanished by the time PLAYSTATE=LOST is logged, but the snapshot preserves what they had). If the player wins, the snapshot from final combat is still accurate (all those minions likely survived).

Programmatically, you could also directly trigger on the end-of-game: if `PLAYSTATE` for the player is LOST or WON, and if WON, just read all entities with `player=<player_id> zone=PLAY`; if LOST, use the last combat snapshot as described.

**Analytics Use:** Final warband composition is useful for understanding endgame strategies. You can display the final board for each game (like a “screenshot” in text form of minions and their stats). Over multiple games, this data could show which minions or builds the player tends to end on (e.g. “Most common final warband tribes: Mechs 40%, Beasts 30%…”). It also helps evaluate if the player often dies with unused board space or weak minions. Visualization might include showing the minion card images or names for the final board of each match, or summarizing the counts of each minion across final boards. Developers building a replay or summary tool often include a final board state – for example, _Hearthstone Deck Tracker’s Battlegrounds overlay_ shows the last known board for opponents as well ([Suggestion for Bob's Buddy (Battlegrounds) · Issue #4109 · HearthSim/Hearthstone-Deck-Tracker · GitHub](https://github.com/HearthSim/Hearthstone-Deck-Tracker/issues/4109#:~:text=The%20last%20known%20board%20state,is%20better%20than%20another%20option)), and one can similarly show the player’s final board.

**Note:** One limitation is that the log will not outright label “this is the final board”. We infer it from timing. Thus, edge cases like conceding a Battlegrounds match early might complicate things (if a player concedes, you might not get a normal combat end; however, conceding in BG still eliminates the player and should set PLAYSTATE=LOST and a placement). Also, if the player dies to a ghost (no opponent), the combat still happens but with an AI copy – the log is the same though. As a parser developer, you should test a few games to ensure your method captures the board correctly in all end conditions. Another quirk: tokens summoned during combat (e.g. by Deathrattles in the final fight) might technically be on the board when the game ends (if you won with summoned minions surviving). Depending on your needs, you might consider the pre-combat board as the final “built” warband, excluding combat-generated tokens. This is a design decision for analytics.

## Constructed Mode Specific Data

In constructed formats (Standard, Wild, Arena, Duels, etc.), the gameplay data revolves around decks and classes rather than economy and shop events. Besides wins/losses and hero classes (already covered), useful data includes the deck used by the player, the opponent’s class, and timings such as game start and end events.

### Deck Used (Player’s Deck Composition or ID)

**Log Source:** _Power.log_ (partial info)  
**Key Patterns:** Hearthstone does **not** explicitly list your 30-card deck in the log file for constructed matches. The client knows your deck (since you selected it), but for security, the log only reveals cards as they are drawn or played. Thus, there is no single log entry for “deck list”. However, you can infer or rebuild much of the deck by tracking card draws and cards remaining. The Power.log will show each card that enters the player’s hand (drawn from deck) or is played from deck. Unknown cards in deck are often referenced with an entity ID but without a card ID until drawn (sometimes they appear as cardID “UNKNOWN” or simply not shown). For example, during mulligan, the log will list the initial cards drawn – at that point those specific cards from your deck become known.

**Extraction:** For a backend parser, fully reconstructing the deck is complex and may be unnecessary. Instead, two practical approaches:

1. **Identify the deck by an external ID** – If your application knows which deck was selected (say the user can provide a deck code or an ID from their collection), you can marry that with the match by using the hero class and cards seen as validation. The logs alone can’t confirm the exact list if some cards never appeared.
    
2. **Collect all cards seen** – You can list every unique card the player drew or played. This will usually be most of the deck, especially in longer games. For missing cards (those never drawn), there’s no log entry, so you’ll have gaps. You might try to deduce them by elimination (if you know the deck size and which cards were revealed, the rest are the unknowns). In practice, many trackers avoid this deduction and instead rely on the user importing their deck list.
    

For actionable analytics, you might not need the full 30-card list. Often it's enough to know the **deck archetype** or a short name (e.g. “Secret Mage” or “Pirate Warrior”). This can be determined by looking at key cards that were played. Some tools use internal rules or machine learning to classify the deck archetype from the partial list of cards seen.

**Analytics Use:** Knowing the deck (or at least archetype) enables far richer analysis: win rates by deck, performance against specific opposing decks, and tracking tech card usage. If you can identify the deck, you might present stats like “Games played with _Pure Paladin_: 10 wins, 5 losses”. Even without full info, listing the significant cards played can be a useful summary for the user (“MVP cards drawn in this game”). For visualization, you might show the card list as known (perhaps grey-out unknown cards if some remained hidden). If deck import is in scope, you could even show the full deck and highlight which cards were drawn or not drawn in that match.

**Note:** This is one area with limitations – if verbose logging isn’t enabled, you still see cards drawn, but if the opponent’s deck is being considered, note that you **will not** see their unknown cards at all until played. So reconstructing the opponent’s deck is even harder (only possible via guesswork or external data). Also, if the user is using a deck tracker in parallel, that tool might supply the deck list to your backend (outside the scope of log parsing). For pure log parsing, accept that “deck used” might be limited to an identifier or partial info. Always consider privacy and TOS: reading the full deck from memory is something official deck trackers do with Blizzard’s blessing, but sticking to logs only, you’re a bit restricted.

### Opponent Class

**Log Source:** _Power.log_  
**Key Patterns:** The opponent’s class is determined by their hero, which is readily available as discussed. In the log, once you identify the opponent’s hero cardID, you can map that to the class (e.g. `HERO_01` is Garrosh – Warrior). There may also be a plaintext indication: often the log line creating the opponent’s hero includes the hero’s name. For example, `FULL_ENTITY - Creating [id=3 cardID=HERO_01 name=Garrosh Hellscream]`. If verbose, you get the name; otherwise you at least have HERO_01. Additionally, the beginning of the match (if you have logging on **LoadingScreen.log** or similar) might say “Versus Warrior” in UI terms, but usually that’s not needed since we have the hero.

**Extraction:** Use the same method as for the player’s hero, but for the opponent’s entity. During `CREATE_GAME`, two players are listed. You can track which one is not you, and extract their hero card. By the time mulligan is over, you will know the opponent’s hero cardID with certainty. Store that and map to class. This is straightforward once the mapping is set up.

**Analytics Use:** Opponent class is fundamental for matchup statistics. It allows grouping of results by what you faced. A common visualization is a matrix or table of your win-rate against each class. For example, “Mage: 70% win rate (7-3), Warrior: 40% (2-3), etc.”. It can also feed into per-deck analysis (maybe your Control Shaman deck has a 80% win rate vs. Warriors but only 50% vs. Mages). Such insights can inform strategy or deck-building choices. In real-time, some deck trackers even display the opponent’s class and deck archetype once enough cards are seen – purely from logs, you can’t confirm archetype, but class is immediate. Your tool can at least label each game’s entry with the opponent’s class.

**Note:** There are nine basic classes (and possibly Demon Hunter, Death Knight as additional classes introduced after initial nine). Ensure your mapping covers new heroes (e.g. Illidan for Demon Hunter, and the Death Knight heroes). If Blizzard adds new classes or heroes, their cardIDs would need an update in your mapping. The log provides the raw data, but your code’s interpretation must stay updated with game content.

### Game Start and End Events

**Log Source:** _Power.log_ (and possibly a bit from other logs like _Asset.log_ for rank-up animation, but Power.log suffices)  
**Key Patterns:** **Game start** is indicated by the `CREATE_GAME` block in Power.log. When a new match begins, you will see something like:

```
CREATE_GAME
    GameEntity EntityID=1 ...
    Player EntityID=2 PlayerID=1 name=...
    Player EntityID=3 PlayerID=2 name=...
```

This is typically the first few lines of a match’s log section. A parser should use this to recognize a new game instance. Track-o-Bot’s parser, for example, registers a handler for the "CREATE_GAME" line to reset state for a new match ([track-o-bot/src/HearthstoneLogTracker.cpp at master · stevschmid/track-o-bot · GitHub](https://github.com/stevschmid/track-o-bot/blob/master/src/HearthstoneLogTracker.cpp#:~:text=RegisterHearthstoneLogLineHandler%28%20,)). After that, mulligan choices are logged (via `GameState.DebugPrintEntityChoices` entries), which also provide player names and IDs. The **hearthstone-log-watcher** signals a `game-start` event once it has seen enough to identify which player is local ([hearthstone-log-watcher - npm](https://www.npmjs.com/package/hearthstone-log-watcher/v/1.0.1#:~:text=game)).

**Game end** can be determined by the presence of the final outcome tags (PLAYSTATE changes to WON/LOST/TIED as discussed). Additionally, there is often a tag update on the **Game entity** itself: the game entity’s `STATE` tag might go to `COMPLETE`. In some logs you might find `TAG_CHANGE Entity=GameEntity tag=STATE value=COMPLETE` at the very end. Not all tools rely on that; it’s safe to consider the game ended once all players have left the PLAYING state. Another indicator in ranked play is the appearance of rank results in other logs (the **Bob.log** oddly prints rank updates for Ranked mode; e.g. a line containing “legend rank ” appears when you hit legend or change rank ([track-o-bot/src/HearthstoneLogTracker.cpp at master · stevschmid/track-o-bot · GitHub](https://github.com/stevschmid/track-o-bot/blob/master/src/HearthstoneLogTracker.cpp#:~:text=RegisterHearthstoneLogLineHandler%28%20,%26HearthstoneLogTracker%3A%3AOnLegendRank))). But for a general solution, focusing on Power.log is enough.

**Extraction:** Implement a state machine: when you see `CREATE_GAME`, start a new record (initialize all data structures for tracking that match). Record the timestamp (the log timestamp on that line is the game start time). When you see the game-ending conditions (either the local player’s PLAYSTATE or both players’ PLAYSTATE no longer `PLAYING`), mark the game as ended and finalize the record. You can also capture the end timestamp from the log. Calculating the duration is as easy as end time minus start time (the log times are in the format `D HH:MM:SS.millisecond`).

It’s wise to also handle abnormal ends: if a player disconnects or quits, the log might show `PLAYSTATE value=CONCEDED` or `QUIT`. Those should be treated as game end as well (with a loss for the quitter). A tie shows `TIED`. Design your parser to classify those appropriately (e.g. concession is essentially a loss outcome).

**Analytics Use:** Capturing game boundaries is essential for slicing the log into individual match data. It lets you output one set of stats per game. The start time and end time give match duration, which can be analyzed (average game length, etc.). You might chart game lengths or see if certain matchups run longer. Also, knowing the end of a game is when you’d finalize all counters for that game (e.g. total turns can be counted by the last turn number tag you see – `TAG_CHANGE tag=TURN` increments each turn). Many visuals like timeline of actions per turn, or a summary of the match, all rely on knowing where the match starts and stops in the log. Without correctly splitting by games, data from consecutive matches would jumble together.

**Note:** One quirk is that if the user remains in the Hearthstone client, the Power.log is continuous – game after game is appended. So your parser must separate games by these events. Also, if the user plays modes like the single-player adventures, those also produce Power.log entries that might need filtering out if you only care about PvP games. The `CREATE_GAME` entry includes a game mode or mission ID sometimes, but not always clearly. If needed, you can use the presence of Bob.log activity to infer a Battlegrounds game versus a constructed game (BG will have Bob.log entries and multiple players >2). For standard vs. arena etc., it might not be distinguished in log, but the deck tracker community uses heuristics (like deck ID ranges, or reading `LoadingScreen.log`). For most analytics, just treating any two-player game as a constructed match and eight-player as BG is sufficient.

## Implementation Considerations and Limitations

Parsing Hearthstone logs for analytics is powerful but comes with some practical considerations:

- **Enabling Verbose Logging:** By default, Hearthstone’s logs are minimal. To get all the data discussed, users must enable verbose logging for the relevant categories. This is done by creating/editing a `log.config` file in the Hearthstone directory. At minimum, `[Power]` (game state) and `[Bob]` (Battlegrounds) should have `LogLevel=Verbose` and `FilePrinting=true`. Without this, crucial events (card draws, tag changes, shop details) will not appear, and your parser will have incomplete data.
    
- **Performance and Size:** Verbose logs can grow large and may slightly affect game performance (framerate dips were noted by some when Power.log is verbose ([Simple Hearthstone logging - see your complete play history without ...](https://www.reddit.com/r/hearthstone/comments/268fkk/simple_hearthstone_logging_see_your_complete_play/#:~:text=Simple%20Hearthstone%20logging%20,Navigate%20to%20your%20Local))). If building a tool for many users, be mindful of log size management – you may need to truncate or rotate logs to avoid huge files. Your parser should ideally stream-process the log (read new lines as they append) rather than load entire multi-MB files into memory at once.
    
- **Duplicate and Out-of-Order Entries:** As noted, the Power.log can log the same game event twice from two sources (GameState vs PowerTaskList) ([hearthstone-battlegrounds-simulator/src/log_parser.cpp at master · twanvl/hearthstone-battlegrounds-simulator · GitHub](https://github.com/twanvl/hearthstone-battlegrounds-simulator/blob/master/src/log_parser.cpp#:~:text=%2F%2F%20log%20line%20type)). This can confuse naive parsers (you might count things twice). A robust parser will either ignore one source or de-duplicate events. Also, if reading both Power.log and Bob.log, be aware they are separate files. Their timestamps can be used to order events between them if needed. Usually, it’s clear (Bob.log covers shop, Power.log covers combat), but if you need a unified timeline, merge by timestamp.
    
- **Timing Issues:** The transition between phases can cause some slight ambiguities – e.g., a reroll might be logged in Bob.log just before the gold change in Power.log. If your analysis correlates these, you’ll need to synchronize carefully. Another timing quirk is that the final tags (like PLAYSTATE or LEADERBOARD_PLACE) might appear _after_ some other end-of-game animations. Always wait for those final tags before concluding the game outcome.
    
- **Missing Data:** Some data simply isn’t in the logs. For example, **MMR changes** in Battlegrounds or Arena rewards are not logged in Power/Bob logs ([Can not find hero data in hearthstone battleground log file - Technical Support - Hearthstone Forums](https://us.forums.blizzard.com/en/hearthstone/t/can-not-find-hero-data-in-hearthstone-battleground-log-file/25592#:~:text=,mmr)). As a developer, you may have to omit those or find alternative approaches (like screen reading or Blizzard’s APIs, if available). Additionally, **card identities** that were never revealed remain unknown. So an opponent might have had a card in hand that they never played – you won’t know what it was from the log. Accept that some uncertainty is inherent unless you supplement with external data.
    
- **Entity IDs and Mapping:** The logs use a lot of internal IDs for entities, zones, etc. Building an analytics tool means translating those into human-friendly terms (card names, hero names, etc.). Maintain up-to-date card data – Blizzard’s card library or the Hearthstone API can help map card IDs to names. This is important because new expansions add cards and you want your tool to recognize them.
    
- **Multi-threaded Log Writes:** Hearthstone writes to multiple log files concurrently (Power, Bob, Rachelle, etc.). If you watch them in real-time, ensure your file watchers are asynchronous or handle concurrency. Also, sometimes a single logical event spans multiple lines (especially the multiline `FULL_ENTITY` blocks). Make sure your parser aggregates lines as needed (for example, `FULL_ENTITY` lines might continue with tag lines indented on following lines).
    
- **Testing and Edge Cases:** Be sure to test your parser with different game scenarios: concede games, tie games, extremely short games, etc., to see how the logs differ. For Battlegrounds, test with a top-4 finish vs. bottom-4 to ensure placement is captured. Check if an anomaly (special event rule) changes log patterns. The Hearthstone developers occasionally change the log format or add new tags, so keep an eye on patch notes – but generally, the core format has remained consistent.
    

By keeping these considerations in mind, a developer can build a robust CLI or backend service that parses Hearthstone log files and transforms them into meaningful, structured data. With that data, the sky’s the limit for analytics and visualization – from simple win rate trackers to detailed turn-by-turn replays complete with card play sequences and resource graphs. The combination of **Power.log** and **Bob.log** provides a comprehensive view of gameplay that, when harnessed properly, yields actionable insights for players and an informative foundation for any Hearthstone analytics tool.
